# ADR-0010: 正規IP空間と写像としてのIP名寄せ

- **Date**: 2026-03-21
- **Status**: Proposed

## Context

イベントタイトルからIPを抽出する際、HaikuがIPを自由記述で返すため表記ゆれが発生し、同一IPが複数エントリとしてip_registryに登録されてしまう問題があった（例: `あんさんぶるスターズ!!` と `Ensemble Stars!!` が別エントリになる）。

ip_registry自体に名寄せの「正解」が存在しないまま名寄せしようとしていたことが根本原因。

## Decision

### 正規IP空間の定義

IP名寄せの対象（dst）を、信頼できる外部ソースが定義するエンティティの集合として定義する。

```
正規IP空間 = 信頼できるソース（AniList, Holodexなど）が定義するエンティティの集合
```

ソース別の正規空間候補:

| ジャンル | 正規ソース | エンティティ |
|----------|------------|-------------|
| アニメ/マンガ | AniList | media ID + 正規タイトル |
| VTuber | Holodex | channel ID + 正規名 |
| ゲーム | TBD | - |

### IP名寄せの再定義

```
写像: イベントタイトル → 正規IP空間のエンティティ
```

ip_registry = この写像の結果を蓄積したもの。**実際にイベントが観測されたIPのみ登録される**（正規ソースの全IPを事前に取り込まない）。

### コスト設計

- 正規IP空間はソースから「全取り込み」しない
- 写像はイベント処理のタイミングでオンデマンドに実行
- ip_registryに登録されるのは「イベントで見かけたIP」のみ → 活動中IPだけが自然に蓄積される

## Rationale

### なぜIP辞書を事前に全取り込みしないか

過去に `whitelist.json` ベースでIP辞書を持っていた時代、イベントが全くないIPもリストに含まれ、それを追跡するコストがかかった。この反省から「アクティブIPのみを追う」方針になった。

正規空間を「事前に全ロード」ではなく「写像の正解セット」として使うことで、コストをかけずに名寄せの一貫性を保てる。

### 写像の設計課題

写像関数（Haiku）に何を返させるかが次の設計課題:
- 正規名（文字列）: 実装シンプル。ただし正規ソースとの同期が必要
- 正規ID（数値ID）: 一意性が高い。APIルックアップが必要

詳細はADR-0011以降で決定する。

## Consequences

- ip_registryの各エントリは将来的に `canonical_source` + `canonical_id` を持つ設計になる
- extract_ip.pyの写像ロジックを正規空間への参照に変更する
- 正規ソースが対応していないジャンル（ゲームなど）は別途設計が必要
