# ADR-0003: スクレイピングとAIの役割を明確に分離する

- **Status**: Accepted
- **Date**: 2026-03-21

## Context

初期実装は公式サイトからのイベント抽出を Playwright + `claude -p` に全面依存していた。
構造化サイトがほぼ存在しない（これが参入障壁でもある）ため、AIによる非構造データの構造化は必要。

しかし全フィールドをAIに任せると：
- APIコストが高い（IPごとに複数ページ × claude -p）
- 失敗時の原因特定が難しい（スクレイピング失敗 vs AI抽出失敗の区別がつかない）
- Hallucination リスク（存在しない価格・日付が生成される）
- グラウンディングが取れない（生データが残らない）

## Decision

「raw取得（スクレイピング）→ 構造化フィールドは直接パース → 残りだけAI」の段階的フィルタを採用。

| フィールド | 手段 | 理由 |
|---|---|---|
| タイトル・URL | スクレイピング | 必ずラベルがある |
| 日時・会場・価格（チケットサイト） | スクレイピング | 構造化済み |
| IPタグ・content_type分類 | AIバッチ | タイトルからの判断が必要 |
| 日時の非標準表記 | AI | 「3月上旬」「来春」等 |
| 公式サイトからのイベント抽出 | Agent（最後の手段） | 非構造HTML |

グラウンディング設計：
- `source_url` + `raw_*` フィールドを必ず保存
- AI抽出フィールドには `extracted_by: "ai"` + `confidence` を付与
- 生データ（`event_source_record`）は削除しない

コスト最適化：
- AIへのリクエストはバッチ化（1件ずつ叩かない）
- 分類タスクは安いモデルで十分

## Consequences

- `event_source_record` テーブルに raw_* フィールドが必要
- AIの失敗と構造化の失敗が分離して追跡できる
- 後からパーサーを改善すれば、生データを再処理できる
