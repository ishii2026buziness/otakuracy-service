# ADR-0011: 正規IP空間のソース選定

- **Date**: 2026-03-21
- **Status**: Accepted

## Context

ADR-0010で正規IP空間の概念を定義した。本ADRでは「どのソースを正規IP空間のベースとして採用するか」をジャンル別に決定する。

権威ソースが満たすべき性質:

- **安定したID**: 同一エントリが常に同じIDで参照できる
- **正規名称**: 表記ゆれの「正解」として使える公式名が取れる
- **機械可読API**: プログラムから照合・取得できる
- **コミュニティ認知**: そのジャンルで「ここが正しい」という共通認識がある

## Decision

### 採用ソース

| ジャンル | 権威ソース | 正規ID | 正規名 |
|----------|------------|--------|--------|
| アニメ/マンガ | AniList | `media.id` | `title.native` |
| VTuber | Holodex | `channel.id` (= YouTube channel ID) | `channel.name` |

### ゲームの除外

ゲームジャンルはイベント数が限定的なため、正規空間の対象外とする。将来的にゲーム関連イベントの観測が増えた場合に別途ADRで対応する。

## Rationale

### AniList（アニメ/マンガ）

- GraphQL APIで `media.id`（整数）と `title.native`（日本語原題）が取得可能
- IDは削除・変更されにくく安定している
- `title.native` は原語タイトルを返すため、日本作品であれば日本語名が得られる
- AniListはアニメ・マンガを同一 `Media` エンティティで管理しており、単一の正規空間として扱える

### Holodex（VTuber）

- `channel.id` はYouTube channel IDと同一であり、Googleが管理する永続的なID
- VTuber専門の集約サービスとしてコミュニティ認知が高い
- channel IDベースのため、Holodexが廃止されてもYouTube側でIDは有効

## Consequences

- ip_registryの `canonical_source` フィールドは `"anilist"` または `"holodex"` を値として持つ
- ip_registryの `canonical_id` フィールドはそれぞれの正規IDを格納する
- 写像（Haiku）はイベント処理時にオンデマンドにAniList/Holodex APIを参照して正規IDを解決する
- ゲーム・その他ジャンルのIPはイベントで観測されても正規IDなしで登録される（`canonical_source = null`）

---

## 付録: 写像パイプラインの設計方針

正規IP空間への写像（イベント観測 → canonical_id解決）の設計方針を記録する。

### 問題の本質

「あんスタ」→「あんさんぶるスターズ!!」のような写像は、**名前→名前の文字列マッチングでは解けない**。
略称はコミュニティが形成した固有知識であり、アルゴリズムでは導出できない。

### 空間変換による解決

写像困難な「表層文字列空間」から「文脈・説明文空間」へ変換することで問題を簡略化する。

```
イベントサイトからの取得情報（文脈セット）
  ├── イベントタイトル
  ├── 出演者名
  ├── 会場
  └── イベント内容・説明文
          ↓ SoftMatcha（word embedding ベース fuzzy grep）
正規IP辞書の説明文空間で類似検索
  └── 候補 Top-N に絞り込み
          ↓ LLM + web検索
「この候補の中でどれが正しい名寄せ先か？」を discriminative に判定
```

**ポイント:**
- 出演者名・主催会社名は略されないことが多く、文脈のアンカーになる
- LLMには候補セットを渡して選ばせる（生成ではなく選択）→ hallucination 対策
- web検索でLLMの判断を外部情報で補強する
- 正規IP辞書の説明文はAniList/Holodexから取得するため自動で整備される
- 辞書空間（AniList→別ソース）を切り替えても写像ロジックは変わらない

### スモールスタート方針

まずLLMのweb検索能力のみで名寄せを検証する。

現在のextract_ip.pyはClaude GatewayでHaikuに**タイトル文字列のみ**を渡しており、web検索は使っていない。
最初の改善として、イベントの文脈情報（出演者・会場等）を含めたプロンプト + web検索を有効化し、精度を計測する。

SoftMatcha導入（候補絞り込みレイヤー）はその後、LLMのみでは解決できないケースが確認されてから判断する。

### 将来構成（SoftMatcha導入後）

正規IP辞書に説明文を付与し、SoftMatchaでインデックス化。イベント文脈をクエリとして候補Top-Nを生成後、LLM + web検索で最終判定。
